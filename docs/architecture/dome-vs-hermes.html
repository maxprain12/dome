Dome vs Hermes — arquitectura de harnesses de agentes

Comparación de los dos harnesses obviando el sistema de gateways de Hermes (Telegram/Discord/Slack/email/WhatsApp/SMS). El objetivo es entender la forma de cada sistema: bucle, contexto, memoria, sesiones, tools, skills, compactación, scheduling y orquestación. Si conoces uno, el otro se entiende como un conjunto de variantes.

Resumen en una página

Ambos harnesses resuelven el mismo problema: dar a un LLM un bucle persistente con memoria, contexto, tools y scheduling. Lo hacen con filosofías distintas — Hermes apuesta por la simplicidad textual (Markdown como fuente de verdad), Dome por la orquestación rica (DAG, sub-agentes, HITL, IPC). Esta página te ayuda a saltar entre los dos.

Dome Electron desktop app

Runtime único @dome/agent-core + LLM connector @dome/ai. Bucle stream → tools → repeat con hooks beforeToolCall (caps + HITL) y context (compactación por resumen). Persistencia de sesión en JSONL, runs en SQLite, sub-agentes anidados (task, delegate_to_agent), workflows como DAG topológico, threads/time-travel.

7 superficies: many, agent-chat, workflows, agent-team, subagent, agent-team-member, threads. Frontend React con tabs; main process en CommonJS.

Hermes Always-on agent

CLI + gateways (Telegram/Discord/Slack/email/SMS/WhatsApp) + API. Bucle clásico: recibe → contexto → LLM → tools → respuesta → memoria. Persistencia en SQLite por sesión, memoria principal en tres archivos Markdown (soul.md, user.md, memory.md). Cron propio con jobs.json. Memoria externa opcional (mem0, SuperMemory).

1 superficie principal (loop), una por gateway. Sin sub-agentes anidados, sin HITL explícito, sin DAG, sin time-travel. La potencia viene de la composición externa (vía gateways).

Cómo está organizada esta página

  • Matriz 16-d: una tabla compacta que cruza cada dimensión y da un veredicto (Dome mejor / Hermes mejor / equivalente / único de Dome / único de Hermes / diferente).
  • Bucle: la pieza central de cualquier agente. Aquí se ve cómo cada uno hace stream → tools → repeat y qué metadata persiste por turno.
  • Memoria + contexto: tres archivos Markdown vs JSONL+SQLite+múltiples memorias, y cómo se combinan con el system prompt.
  • Tools, skills, HITL: catálogo plano, caps por herramienta, system prompt on-demand, aprobación humana.

Bucle del agente

El bucle es la pieza común. La diferencia está en los hooks y en lo que persiste cada turno.

Dome — bucle nativo

El AgentHarness de @dome/agent-core ejecuta prompt(userPrompt, images) y emite AgentEvents que el bridge mapea a chunks del renderer.

// electron/agents/agent-runtime.cjs
const setup = await setupHarness(surface, opts);
const assistant = await harness.prompt(
  userPrompt,
  promptImages.length > 0 ? { images: promptImages } : undefined,
);

Hook beforeToolCall (caps + HITL) y hook context (compactación por resumen) corren entre eventos.

Límite duro: 25 turnos (DEFAULT_RECURSION_LIMIT, env DOME_RECURSION_LIMIT) vía buildTurnLimiter.

Hermes — bucle clásico

El loop es lo que describe el tutorial: recibe → contexto → LLM → tool calls → resultado → respuesta → memoria.

1. User sends a message
2. Hermes builds the context for the request
3. The context and message history are sent to the LLM
4. The LLM may decide to call tools
5. Tool results are sent back to the LLM
6. The LLM produces a final response
7. Hermes updates memory when there is something useful to remember

El último paso — update memory — es la pieza clave: el agente mejora con el uso.

Memoria + contexto

Cómo cada harness decide qué va al contexto de cada turno.

Fuentes de memoria

Dome

  • Markdown legacy (~/.dome/martin/{SOUL,USER,MEMORY}.md) — disociado del runtime nativo, path legacy.
  • Project memory — AGENTS.md en la raíz del proyecto, inyección manual por el caller.
  • Sesiones JSONL — JsonlSessionRepo en userData/agent-sessions/. Tree con entries, forks, compactions.
  • Runs SQLite — automation_runs + automation_run_steps + automation_run_links.
  • Knowledge base local — LanceDB (dome-lance) + FTS (DuckDB fts extension) + grafo. RAG vía tool resource_hybrid_search.
  • Tool remember_fact — para que el modelo guarde hechos del usuario.

Hermes

  • Markdown vivo — soul.md (rol), user.md (sobre el usuario, reescrito por el modelo), memory.md (notas durables).
  • SQLite — una fila por interaction con session_id. Identifica el canal (Telegram, Slack, email).
  • Memoria externa opcional — mem0, SuperMemory. Cuando está habilitada, recupera memorias relevantes más allá de los Markdown.
  • Historial reciente — los últimos mensajes de la sesión + descripción de tools + skills.

Compactación

Dome

DEFAULT_COMPACTION_SETTINGS = {
  enabled: true,
  reserveTokens: 16384,
  keepRecentTokens: 20000,
}

shouldCompact(estimate, window, settings) =
  estimate > window - reserveTokens

Genera resumen con generateSummary, lo inserta como primer mensaje del nuevo contexto, persiste un CompactionEntry en el JSONL.

Trigger: automático cuando se cumple el umbral. Manual: threads:compact IPC.

Hermes

Threshold: 50% of model context window
(configurable; lower for small-window models)

Summary preserves:
  - goal of the conversation
  - completed actions
  - active state
  - blockers
  - key decisions
  - resolved questions
  - relevant files
  - previous summaries
  - turns still to incorporate

Hermes define explícitamente la estructura del resumen; Dome define los tokens y el corte. Los dos cubren el mismo problema.

Tools, skills, HITL

Tools

Dome

102 tools en 21 familias, registradas con tipado y timeout por tool. Caps explícitos para evitar abuso.

  • Registry: @dome/tools/src/families.ts + createToolRegistry
  • Dispatcher: tool-dispatcher.cjs con TOOL_HANDLER_MAP → ai-tools-handler.cjs
  • Cap global: 200 tool-calls / run
  • Cap por tool: CREATION_TOOL_CAPS (19 entries) o DEFAULT_PER_TOOL_CAP = 50
  • dome_load_doc: 12 docs on-demand (entity_rules, artifacts, …)
  • Helpers: normalizeToolInput, capToolResultString (48k default), formatToolResultForModel

Hermes

Las tools se describen en el system prompt. El agente decide cuándo invocarlas por nombre.

  • Sin catálogo estructurado público
  • Sin caps explícitos por tool
  • Sin timeout por tool en el código del loop
  • El costo del LLM es el único freno natural
  • El gateway mapea entradas externas al formato interno de la tool

Skills

Ambos exponen skills al agente como descripciones, con cuerpo disponible bajo demanda. La diferencia es de formato.

SKILL.md (Dome) 
---
name: Commit Helper
description: Write clear, conventional git commit messages and summarize a set of changes.
when_to_use: When the user asks for a commit message, wants to summarize a diff, or is preparing to commit code.
---

# Body in markdown...

YAML frontmatter tipifica name, description, when_to_use, opcional disableModelInvocation. Inyección en system prompt como bloque XML:

<available_skills>
  <skill>
    <name>Commit Helper</name>
    <description>...</description>
    <location>/Users/.../SKILL.md</location>
  </skill>
</available_skills>
Skills (Hermes)

Las skills también se inyectan como descripciones en el system prompt; el cuerpo completo está disponible. Formato más libre — no se exige frontmatter YAML.

Sub-agentes, workflows y scheduling

Aquí es donde más difieren. Dome es un motor de orquestación + agente; Hermes es un agente siempre-activo.

Sub-agentes

Dome — dos tools de delegación

task (Many) llama a un sub-harness con surface subagent y tool subset fijo. delegate_to_agent (Agent Team) llama a un member con surface agent-team-member y tools por member.toolIds.

// electron/agents/subagents-native.cjs
const SUBAGENT_NAMES = ['research', 'library', 'writer', 'data'];

const text = await agentRuntime.runAgent('subagent', {
  provider, model, apiKey, baseUrl,
  messages, toolDefinitions: toolDefs,   // subset fijo
  useDirectTools: true, skipHitl: true,
  threadId: `${parent.threadId}_sub_${name}_${ts}`,
  parentThreadId: parent.threadId,
});

Limitación confirmada: task no se inyecta dentro del subagente — sin recursión.

Hermes — sin sub-agentes

El loop es plano. No hay delegation interna. La "composición" se hace a nivel externo: múltiples instancias del agente, o el humano decide qué instancia atender qué tarea.

Si necesitas "research + writer" tendrías que:

  • Definir un agent research y uno writer como dos procesos.
  • Coordinar entre ellos a nivel de gateway o con código.
  • O el humano encadena turnos.

Workflows / orquestación

Dome — DAG topológico

Workflows son nodos conectados por edges. topologicalLevels() agrupa por dependencias; intra-nivel se ejecuta en paralelo (Promise.all); entre niveles, en serie. Cada nodo-agent pasa por el harness (runAgent('workflows', …)).

// electron/agents/workflow-executor.cjs
const levels = topologicalLevels(wfNodes, wfEdges);
for (const level of levels) {
  const results = await Promise.all(
    level.map((node) => runNodeWithRetry(nodeRunners.get(node.id)))
  );
  for (const r of results) if (r?.payloads) Object.assign(state.payloads, r.payloads);
}

Tipos de nodo: text-input, document, image, output, agent. Retry 3× con backoff exponencial (500ms·2^j + jitter).

Hermes — sin DAG

No hay concepto de workflow. La "orquestación" es la concatenación de turnos que el usuario o un script externo decide.

El cron loop ejecuta jobs aislados; no encadena resultados entre jobs.

Scheduling

Dome — automation-service

// electron/agents/automation-service.cjs
setInterval(tick, 60_000);
// isDue(automation, ts) por triggerType:
//  - 'schedule' (daily, weekly, cron-lite)
//  - 'resource_opened' (contextual, vía notifications IPC)
//  - 'feeder' / 'workflow' / 'agent' / 'many' como targetType

Dispara runs completos. Persiste run en automation_runs, ejecuta con startAutomationNow que decide entre feeder/workflow/agent/many. Si hay automation_artifact_bindings, el resultado se mapea a artefactos persistidos (Kind B) vía artifact-sink.cjs.

Hermes — jobs.json loop

El loop propio lee jobs.json en intervalos y ejecuta los jobs que encuentra. La documentación dice SQLite pero la implementación analizada usa JSON.

Casos típicos:

  • Enviar un email cada mañana con AI news
  • Post diario a Slack
  • Mensaje semanal al jefe
  • Recurring agent tasks

El cron interactúa con gateways: si el job debe notificarte en Telegram/Slack, el gateway debe estar configurado.

Patrones transferibles

Lo que puedes portar de uno al otro.

De Hermes a Dome

  • Memoria Markdown viva: el modelo reescribe user.md/memory.md. Dome tiene el path legacy disociado; revivirlo como mecanismo de aprendizaje es lo que más le falta.
  • Estructura explícita del resumen: objetivo, acciones, blockers, decisiones, archivos relevantes. Dome podría adoptarlo en generateSummary con un prompt estructurado.
  • Concatenación minimalista: el context builder de Hermes es trivial (append). Dome podría ganar un modo "minimal" para casos simples sin perder el assembler rico.
  • Memoria externa opcional: mem0/SuperMemory como providers pluggables. Dome tiene la KB local (LanceDB); añadir recall semántico sobre memorias del usuario es una pieza similar.

De Dome a Hermes

  • HITL explícito: el beforeToolCall con caps + interrupt. Para un always-on agent que toca shell_exec o email_send, la aprobación humana por tool es valiosa. Hermes no la tiene; añadirla es la mejora más obvia.
  • Caps por tool: 200 global + cap por tool evita loops infinitos y abuso. Un siempre-activo se beneficia de límites duros.
  • Sub-agentes / delegación: Hermes es plano. Añadir task(subagent_type, prompt) con un subset de tools (research, writer, data) lo haría más potente sin cambiar el loop principal.
  • Time-travel / fork: JSONL con fork() permite explorar variantes de un thread. SQLite plano no lo soporta; el coste es bajo si se opta por JSONL per-thread.
  • Run como entidad auditable: el RunLogView de Dome con runs:updated + runs:step es un patrón útil para depurar y facturar.
  • Tool registry tipado: createToolRegistry + executeToolInMain desacopla definición de ejecución. Un patrón limpio si Hermes quiere migrar a TS.

Lo que ambos comparten (la columna vertebral)

  • Loop: usuario → contexto → LLM → tools → respuesta → memoria.
  • Context builder: system prompt + historial + tools/skills.
  • Persistencia de sesión: una unidad de conversación con su session_id.
  • Compactación por resumen cuando el contexto se acerca al límite.
  • Scheduling como loop independiente del loop principal.
  • Punto de entrada único: cada harness tiene una función a la que todo converge.

Resumen en una frase

Hermes es la versión minimalista, text-first, donde el modelo reescribe su propia memoria. Dome es la versión orquestada, con HITL, sub-agentes, DAG, time-travel y auditoría por run. El poder viene del mismo lugar: un loop limpio, contexto bien definido, memoria fiable, e integración con los canales por los que la gente realmente se comunica.
