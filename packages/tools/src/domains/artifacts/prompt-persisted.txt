# Persisted artifacts — full state & persistence contract

Companion to the `artifacts` doc. Load before `artifact_create` / `artifact_update_state` / `artifact_merge_data` / `artifact_delete`.

---

## State model

Each artifact stores one JSON `state` in SQLite (mirrored to a vault `.html` file):

```json
{
  "html": "<style>…</style><div id=\"app\">…</div><script>…</script>",
  "css": "optional extra stylesheet",
  "data": { "everything the user can change lives here" },
  "linkedData": null
}
```

- **`html`** — BODY FRAGMENT only (no doctype/html/head/body wrappers — they get stripped). This is the UI + logic. It must RENDER FROM DATA: read `window.DOME_DATA` on load and draw the UI from it.
- **`data`** — the single source of truth for mutable content. Injected as `window.DOME_DATA` before your scripts run.
- **`linkedData`** — read-only snapshot Dome injects when a spreadsheet is linked via `artifact_link_resource`. Render it; never write to it.

## The golden pattern: data-driven rendering

```
<style>/* tokens only */</style>
<div id="app"></div>
<script>
  var state = window.DOME_DATA || {};
  state.items = Array.isArray(state.items) ? state.items : [];

  function save() { window.__dome_updateState(state); }
  function render() { /* rebuild #app from state — including the empty state */ }

  // every mutation: update state → save() → render()
  render();
</script>
```

Why: Dome re-injects `data` on every open, pushes external updates (agent patches, Excel refresh) via `window.dome_onDataRefresh(next)` if you define it, and the toolbar Save button calls your optional `window.__dome_collectState()`. If your HTML hardcodes rows instead of rendering from data, all of that silently breaks.

**Persistence rules**

- Call `__dome_updateState(state)` after EVERY meaningful mutation (add row, toggle, reorder, text blur). It writes to SQLite immediately.
- Never rely on `localStorage`/`sessionStorage` — a shim keeps legacy code from crashing but nothing written there survives.
- Inputs you want auto-mirrored get `data-dome-key="fieldName"` (or `id`/`name`); Dome copies their values in/out of `DOME_DATA` automatically. For anything richer (arrays, objects) manage the state object yourself.

## Lifecycle & tools

| Tool | Use it to |
|---|---|
| `artifact_design` | Generate a Dome-themed tabbed dossier (`html` + seed `data`) for read-heavy layouts. Not persistence — pass the result into `artifact_create`. Load `dome_load_doc('artifact_design')` first. |
| `artifact_create` | Create: `{ artifact_type: 'task-tracker'\|'chart'\|'custom', html, data?, title?, project_id? }`. `title` optional (falls back to the fragment's first heading / type). |
| `artifact_get` | Read current `state` + metadata before editing an existing artifact. ALWAYS get before update. |
| `artifact_merge_data` | Shallow-merge keys into `state.data` WITHOUT touching html: `{ resource_id, data_patch }`. The default choice for pushing rows/KPIs after `excel_get`, `resource_get`, web research. |
| `artifact_update_state` | Replace `html` and/or `data`: `{ resource_id, html?, data? }`. Only for redesigns or full data resets — for incremental data prefer `artifact_merge_data`. |
| `artifact_list` | Enumerate artifacts in the project (ids, titles, types). |
| `artifact_delete` | Remove the artifact resource (asks nothing — confirm with the user first). |
| `artifact_link_resource` | Bind a spreadsheet: `{ artifact_resource_id, linked_resource_id }` (null unlinks). Dome then refreshes `linkedData` on every open/edit and shows a Refresh button. |

**Concurrency rule:** data-only changes go through `artifact_merge_data` (or `artifact_update_state` with only `data`) — these merge server-side into the CURRENT state, so a user editing the artifact at the same time never loses html/css.

## Data placement: `state.data` vs hardcoded HTML

| Situation | Where it goes |
|---|---|
| User will add/edit/toggle between sessions | `state.data`, rendered by your JS |
| Bulk rows from Excel/PDF/research after creation | `artifact_merge_data` patch — NEVER paste megatables as static HTML strings |
| Fixed labels, section titles, legends | Hardcoded in the html fragment is fine |
| Always-fresh spreadsheet grid | `artifact_link_resource` → render `linkedData` read-only; user overrides go in `data` |

**Excel → dashboard (canonical flow)**
1. `excel_get(resource_id)` per sheet → understand schema, compute KPIs.
2. `artifact_create` with html that renders `window.DOME_DATA.rows` + `data: { rows: firstRows }` (≤50 seed rows).
3. `artifact_link_resource({ artifact_resource_id, linked_resource_id })` → live grid via `linkedData`.
4. User filters/notes persist via `__dome_updateState` inside the iframe.

Soft limit: keep `html` under ~200 KB. If you are tempted to inline data to get past that, the data belongs in `state.data`.

---

## External / live data (LAN, APIs, shell)

The iframe cannot reach local services. When the artifact needs periodic external data: `dome_load_doc('feeders')` → `feeder_create` (user approves the script) → `feeder_run`, and schedule refreshes with a feeder automation. Never ask the user to paste JSON by hand.

---

## Quality checklist (run mentally before `artifact_create`)

1. Fragment only — no doctype/html/head/body wrappers.
2. Renders entirely from `window.DOME_DATA`; seed `data` provided; empty state designed.
3. Every mutation calls `__dome_updateState`; no browser-storage persistence.
4. Tokens only (`--background`, `--card`, `--primary`, `--foreground`, semantic vars); no gradients; no emoji icons — inline Lucide-style SVG.
5. Self-contained vanilla JS; any CDN use degrades gracefully offline.
6. Usable at 800px width; interactive targets ≥32px; visible focus states.
7. UI copy in the user's conversation language.

After a successful `artifact_create`, emit the `artifact:created_entity` fenced card per `entity_rules` and summarize in natural language — never dump raw JSON at the user.
