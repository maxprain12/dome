### Persisted library mini-apps (`artifact_create`, `artifact_update_state`, …)

Use this companion to `artifacts.txt` whenever you mutate **Kind B — library** artifacts stored in Dome’s SQLite catalog.

---

## Lifecycle

> **Title is auto-derived** — `title` is OPTIONAL in `artifact_create`. If omitted, Dome falls back to: HTML `<title>` tag → `artifact_type` → `'Untitled Artifact'`. Pass it explicitly only when you want a specific display name.

1. **Design** — Before `artifact_create`, call `dome_load_doc('artifact_design')` unless the user asked for a **mini/simple** artifact (one section, no tabs) and context is already sufficient. For mini artifacts: skip `artifact_design` tool; pass compact `html` + seed `data` from conversation or `file_read` results into `artifact_create`. Always call `dome_load_doc('artifact_persisted')` once if not loaded this session.

   For full dashboards, call `dome_load_doc('artifact_design')` and choose one of the two design paths:

   | Artifact kind | Design path |
   |---|---|
   | **Structured / read-heavy** (reports, KPI summaries, research digests, timelines, model cards, multi-section references) | Call `artifact_design` **tool** with a `spec` → get Dome-themed HTML + seed `data` → pass both into `artifact_create`. |
   | **Interactive / stateful** (tables with filters, Kanban boards, forms, charts, dashboards with user edits) | Load the design doc for CSS variable tokens + SVG icon rules, then write **custom HTML** that reads `window.DOME_DATA` and calls `__dome_updateState` → pass into `artifact_create`. |

   In both cases: use **only** injected CSS variables (`--bg`, `--accent`, `--primary-text`, `--border`, etc.), inline SVG icons (Lucide style, no emoji/pictographs), and no remote assets / `localStorage`.
2. **Create** via `artifact_create` with flat tool params: `{ title?, artifact_type, html, data?, project_id? }`. Required: `artifact_type` + `html`. `title` is optional (auto-derived). Link Excel **after** create with `artifact_link_resource` — not as a field on `artifact_create`.
3. **Read** via `artifact_get` / list via Many’s hub or `artifact_list`.
4. **Mutate** — inside the iframe call `__dome_updateState` for user-driven edits; from the agent use `artifact_merge_data` (shallow patch to `state.data` only), `artifact_update_state` (replace/merge `html` and/or `data`), or bind Excel with `artifact_link_resource` for read-only grids in `linkedData`.
5. **External / live data** — when the iframe cannot reach local APIs (Redfish, LAN services, shell): load `dome_load_doc('feeders')`, create an approved **feeder** script (`feeder_create` → user approves → `feeder_run`). Schedule with `target_type: "feeder"` automations for periodic refresh. **Never** ask the user to paste JSON manually.

## `state.data` vs datos “hardcodeados” en el HTML

| Situación | Usar |
| --- | --- |
| El usuario editará filas, notas o toggles entre sesiones | `state.data` + JS que lee `DOME_DATA` y llama `__dome_updateState` |
| Volcar muchas filas desde Excel/PDF/agente después de crear el artefacto | `artifact_merge_data` o `artifact_update_state(state.data)`; **no** pegar mega-tablas como strings estáticos en el HTML |
| Texto/UI que nunca cambia (título estático de sección, leyenda fija) | Hardcoded en el HTML está bien |
| Datos siempre frescos desde un Excel vinculado | `artifact_link_resource` y render desde `linkedData` (solo lectura desde el modelo) |

**Ejemplos:** lista de compras con checkboxes → `data.items[]`. KPI estático “Resumen mensual” como label → HTML. Importar 200 filas de `excel_get` → `artifact_merge_data({ rows: [...] })`. Tablero que solo muestra la hoja vinculada → `linkedData`, sin duplicar en `data` salvo overrides del usuario.

**Pattern: Excel → artifact (preferred)**
1. `excel_get(resource_id)` for each relevant sheet to understand schema + extract KPIs.
2. `artifact_create({ artifact_type: 'custom', html: <compact HTML that reads window.DOME_DATA.rows>, data: { rows: [...firstSheet.slice(0,50)] } })` — HTML renders rows dynamically from `DOME_DATA`; do NOT inline thousands of rows as static strings.
3. `artifact_link_resource({ artifact_resource_id, linked_resource_id })` — Dome populates `state.linkedData` on every open so Excel edits propagate automatically.
4. For user-driven filters/notes: persist via `__dome_updateState({ ...window.DOME_DATA, filter: 'foo' })` inside the iframe.
**NEVER** paste megatable strings into the HTML — bloats the template, slows re-renders, and exhausts model output budgets.

## Payload shape (`state`)

`state` is JSON stored in SQLite. Typical persisted HTML mini-apps embed:

```json
{
  "html": "<section>…minimal markup…</section>",
  "data": {},
  "linkedData": null
}
```

- **`html`** — body markup; keep inputs with `data-dome-key`, `name`, or `id` fields so Dome can mirror values ↔ `window.DOME_DATA`.
- **`data`** — mirrored into `window.DOME_DATA` for editable fields (`input`, `textarea`, `select`). Updated values flow back via `__dome_updateState(next)`.
- **`linkedData`** — optional snapshot injected by Dome when the artifact is linked to an Excel via `artifact_link_resource`. Read-only for the model; display only.

---

## Iframe contract (MANDATORY)

Dome injects before your script bundle:

- `window.DOME_DATA` — initial field bag (merge of `state.data` + `state.linkedData` exposure in workspace UI).
- `window.__dome_updateState(nextDataObject)` — call after every meaningful mutation; persists to SQLite immediately.
- `window.__dome_collectState()` — optional hook for explicit saves (toolbar “Save”).

**Rules**

- No `window.electron`, `fetch`, remote assets, or `localStorage` fallback required — Dome injects an in-memory storage shim.
- Always call `__dome_updateState` after programmatic DOM edits that should survive restarts.
- Keep bundle self-contained; inline JS only.

## Tool reference

| Tool | When |
| --- | --- |
| `artifact_design` | Generate Dome-styled tabbed dossier **HTML** + seed **`data`** (not persisted). Pass returned `html` + `data` into `artifact_create`. Load `dome_load_doc('artifact_design')` before first use. |
| `artifact_create` | User needs a durable mini-app (trackers, forms, dashboards that persist). |
| `artifact_get` | Need current `state` / metadata. |
| `artifact_merge_data` | Shallow patch to `state.data`: `{ resource_id, data_patch }` (p. ej. tras `excel_get`) sin re-enviar HTML. |
| `artifact_update_state` | `{ resource_id, html?, data? }` — cambia UI y/o datos cuando rediseñas o reemplazas el objeto completo. |
| `artifact_delete` | Remove resource + artifact row. |
| `artifact_link_resource` | Bind Excel: `{ artifact_resource_id, linked_resource_id }` (null to unlink) for `linkedData` refresh. |

After `artifact_create`, emit the `artifact:created_entity` fenced card described in `entity_rules` so the Hub UI can render shortcuts.

---

## Linked Excel resources

When an Excel is linked (`artifact_link_resource`), Dome may populate `state.linkedData` with `{ sheet_names, sheet_name, data }` grids. Your HTML should read-only render `linkedData` and persist user tweaks through `state.data`.

- Respect token palette + typography rules from `artifacts.txt`.
- Never echo raw JSON to the user; summarize actions in natural language alongside internal tool calls.
