# Artifacts — operational guide (persisted only)

Load this doc (`dome_load_doc({ id: "artifacts" })`) before calling `artifact_create` or related artifact tools.

## Persisted artifacts only

Dome no longer supports inline `artifact:TYPE` chat blocks. **Every interactive artifact must be created with `artifact_create`** and is stored as:

- A library resource (`resources.type = 'artifact'`)
- A portable HTML file in the project vault (`resources.vault_path`, e.g. `Mi dashboard.html`)
- SQLite rows (`artifacts`, `artifact_runtime_data`) as index/cache for search, automations, and iframe sync

After a successful `artifact_create`, the user opens the artifact from the sidebar or the tool-result card in chat.

---

## Decision matrix

| Signal in the user's request | What to do |
|---|---|
| Informe / dosier navegable, varias pestañas, tarjetas con badges | `dome_load_doc('artifact_design')` + `dome_load_doc('artifact_persisted')` → `artifact_design` → `artifact_create` (`artifact_type: "custom"`) |
| "día a día", tracking, registro persistente, mini-app, herramienta que persista | `artifact_create` (+ `artifact_persisted` doc) |
| One-time dashboard / KPIs without persistence | Still use `artifact_create` — vault HTML is the source of truth |
| Note, summary, document | `resource_create` type=note |
| Presentation / .pptx | `ppt_create` |
| Flashcards | `flashcard_create` |
| Quiz / exam | `generate_quiz` |
| Mindmap | `generate_mindmap` |

**Key question:** "Will the user need to add, edit, or come back to this after the conversation?" → YES → `artifact_create`. Even one-shot panels should be persisted so they live in the vault.

---

## Workflow

1. **Load `dome_load_doc('artifact_persisted')`** — iframe contract, `__dome_updateState`, vault mirror rules.
2. **Multi-tab dossier?** Also load `artifact_design`, call `artifact_design`, pass returned `html` + `data` to `artifact_create`.
3. **Call `artifact_create`** with `{ artifact_type, html, data?, title?, project_id? }`.
4. Tell the user the artifact was saved to the vault/library. Do **not** emit ` ```artifact:TYPE ` fences.

---

## Vault on disk

Each artifact is mirrored to `{projectVaultRoot}/{folder}/{Title}.html` with embedded `dome-artifact-state` JSON. Feeders for that artifact live in `{Title}.dome/feeders/<feederId>/` (script + metadata; secrets stay encrypted in the OS vault, never on disk).

External edits to the HTML file are reconciled by the vault watcher back into SQLite.

---

## Design contract (MANDATORY)

Every artifact MUST use Dome's design tokens. Hardcoded colors, gradients, neon palettes, or gothic fonts are rejected.

**Allowed palette** (CSS variables only):
- Surfaces: `var(--bg)`, `var(--bg-secondary)`, `var(--bg-tertiary)`, `var(--bg-hover)`
- Text: `var(--primary-text)`, `var(--secondary-text)`, `var(--tertiary-text)`
- Borders: `var(--border)`, `var(--border-hover)`
- Accent: `var(--accent)`
- Semantic: success/warning/error/info tokens

**Hard prohibitions:** no gradients, no remote assets, no `localStorage`/`sessionStorage` in persisted iframe HTML, no `@media (prefers-color-scheme)` forks.

See `dome_load_doc('artifact_persisted')` for the full iframe + state contract.
